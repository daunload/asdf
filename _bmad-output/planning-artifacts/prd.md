# Product Requirements Document - natalchart

**Author:** daun
**Date:** 2026-01-21

## Executive Summary

**비전:** 출생(네이탈) 차트와 LLM 해석을 **한 화면에 한 섹션(주제)에 대한 내용**으로 제공해, "GPT보다 나를 잘 안다"고 느끼게 하고, 궁금할 때마다 재방문·유료 전환을 만드는 웹 서비스.  
**차별점:** 기존 운세의 긴 글·스크롤을 한 화면에 한 섹션 분량으로 바꾸고, 무료 4주제로 체험·재방문, 유료 10주제로 과금.  
**대상:** 네이탈·운세에 관심 있는 일반 사용자, 모바일 우선.

## Success Criteria

### User Success

- **자기 이해 체감 극대화:** GPT·Gemini보다 "나를 더 잘 안다"고 느끼게 한다. (실제 정확도보다 **체감 정확도**에 초점.)
- **궁금할 때마다 재방문:** "궁금한 게 생길 때마다 여기서 점을 보고 싶다"는 습관·의존을 만든다.
- **재방문 트리거:** 무료(기본 기질·강점·약점)로 "첫인상"과 "다시 보고 싶음"을 만들고, 유료(금전·연애 등)로 "그다음 궁금함"을 과금 전환으로 이어간다.

### Business Success

- **재방문률 극대화:** 핵심 지표는 재방문률. DAU/MAU·체험 완료 수보다 "다시 들어온 비율"을 최우선.
- **과금 전환:** 무료(기본 기질·강점·약점)로 진입·재방문을 만들고, 금전·연애 등 심화 주제는 **과금 후에만** 확인 가능한 구조로 수익화.

### Technical Success

- **LLM → 사용자 구조 변환:** LLM API 응답을 사용자가 이해할 수 있는 구조(한 장 = 심볼 + 주제별 해석 + CTA)로 변환해 제공. 한 화면에 한 섹션(주제)에 대한 내용을 담는다.
- **출생차트 계산:** 생년월일·시간·장소 기반 네이탈/출생차트 계산 로직이 안정적으로 동작해야 한다.
- **LLM API 연동:** 출생차트·컨텍스트를 LLM에 전달하고, 응답을 파싱·구조화하는 파이프라인이 동작해야 한다.
- **접근 제어:** 무료 구간(기본 기질·강점·약점) vs 유료 구간(금전·연애 등)을 구분하는 권한·노출 로직.

### Measurable Outcomes

- **사용자:** "나를 더 잘 안다" 체감(설문/피드백), 카드 2장 이상 열람 비율, 무료→유료 전환 전 흥미 유지.
- **비즈니스:** 7일·30일 재방문률, **무료 사용자 중 유료 전환률**, 심화 주제별 유료 열람 수.
- **기술:** LLM 응답→구조화된 카드 변환 성공률, 출생차트 계산 정확도·에러율, 무료/유료 구간 노출 정확도.

## Product Scope

### MVP - Minimum Viable Product

- **LLM API 연동:** 출생차트·주제 컨텍스트를 LLM에 전달하고, 응답을 **한 장 카드** 구조로 파싱·노출.
- **출생차트 계산 로직:** 생년월일·시간·장소 기반 네이탈/출생차트 계산.
- **무료(기본) 콘텐츠:** 위 4주제(기본 성격·자아, 첫인상·외부 이미지, 재능·강점, 과제·두려움·성장)를 한 장 카드 형태로 기본 제공. 진입·재방문·"나를 잘 안다" 체감의 기반.
- **유료(심화) 콘텐츠:** 위 10주제(감정·무의식, 사고·소통·학습, 사랑·관계, 연애·결혼, 행동·욕망·분노, 직업·커리어, 돈·자존가치·물질, 변화·위기, 영혼·목적, 인생 사건 타이밍)는 **과금 후에만** 확인 가능. 카드/주제 단위 잠금(lock), 결제·구독 시 해제.
- **한 장 카드 경험:** 한 화면에 한 섹션(주제)에 대한 내용. 심볼 1개 + 해당 주제 해석(핵심 1~2문장) + `[ 다음 카드 ]`.
- **입력 플로우:** 생년월일 → 시간 → 장소(스텝당 한 입력, "시간 모를 때" 최소 대응).
- **과금·접근 제어:** 주제별 무료/유료 구분, 유료 주제 카드 노출 전 결제/구독 상태 검사.

### Growth Features (Post-MVP)

- 심화 주제 확대(일/재물/오늘 등) 및 주제별 과금 옵션 세분화.
- 입력·"시간 모를 때" 플로우 정교화.
- 톤·문구 가이드("~해 보세요" 등)로 체감 품질 극대화.
- 재방문·유료 전환 유도(이메일/푸시, "이번 주 무료 카드" 등).

### Vision (Future)

- 1회 결제·구독·후원·UGC 등 다양한 과금·참여 구조.
- 장기 데이터로 "나를 더 잘 아는" 읽기·개인화.

## User Journeys

### 여정 1: 첫 방문 — 무료 4주제로 "나를 잘 안다" 체감

- **Opening:** SNS·지인 추천으로 접속. "나에 대한 말이 왜 이렇게 맞지?"가 궁금함.
- **Rising Action:** 생년월일 → 시간 → 장소 입력(시간 모르면 "모름" 선택). 출생차트 계산 후 **무료 4주제**(기본 성격·자아, 첫인상, 재능·강점, 과제·성장)를 **한 장 카드**씩 제공. `[ 다음 카드 ]`로 이어서 봄.
- **Climax:** "맞아, 나 그렇지", "이렇게 보이구나" 같은 체감. 4장 안에서 "여기서는 나를 잘 안다"고 느낌.
- **Resolution:** 4장 종료 후, 유료 10주제(연애·금전·직업·타이밍 등) **잠금 카드/목록** 노출. "나중에 궁금해지면 다시 와서 풀어보기" 인상.

**요구사항:** 입력 플로우(생년월일·시간·장소, "시간 모름"), 출생차트 계산, 무료 4주제 카드 생성·노출, `[ 다음 카드 ]` 네비게이션, 무료 종료 후 유료 잠금 안내.

### 여정 2: 유료 전환 — 잠금 주제를 보고 결제

- **Opening:** 무료 4장을 본 뒤 "연애"·"돈"·"직업"이 궁금해짐. 재방문하거나, 첫 방문 직후 유료 안내 화면에서 잠금 목록을 봄.
- **Rising Action:** 유료 10주제 중 일부가 **잠금(자물쇠·블러 등)**으로 보임. 주제별로 "이 카드를 보려면 해금이 필요해요" 문구 + CTA(결제/구독). 결제 플로우 진행.
- **Climax:** 결제 완료 후 해당 주제(들) 해금. 한 장 카드 형태로 연애·금전·직업·타이밍 등 확인. "여기서만 이렇게 구체적으로 나온다"는 체감.
- **Resolution:** 해금된 주제는 재방문 시에도 열람 가능. "궁금할 때마다 여기 들어온다"는 습관의 시작.

**요구사항:** 주제별 잠금/해금, 결제·구독 플로우, 구매 이력 기반 접근 제어, 유료 카드 생성·노출.

### 여정 3: 재방문 — 궁금해질 때마다 다시 들어옴

- **Opening:** 며칠/몇 주 후 "연애가 요즘 왜 이러지?", "돈 쓰는 패턴" 등이 궁금해짐. 북마크·직접 검색으로 재방문.
- **Rising Action:** (이미 가입/로그인된 경우) 세션·계정 인식. **유료 구매 이력 있음:** 해당 주제 카드 재열람 or "오늘의 한 장" 등으로 다시 보기. **무료만 이용:** 유료 잠금 안내 다시 노출, 결제 유도.
- **Climax:** "궁금할 때마다 여기 들어오게" 되는 순간. 재방문률 극대화의 핵심 여정.
- **Resolution:** 재방문률·유료 전환률에 기여. "GPT보다 나를 잘 안다" 인식 강화.

**요구사항:** 로그인·세션, 구매 이력 조회, (선택) "오늘의 카드"·재노출, 무료/유료 경로 분기.

### 여정 4: 예외 — "시간 모를 때" 입력

- **Opening:** 출생 시간을 모름. "시간 모를 때도 볼 수 있나?"라고 기대하며 입력.
- **Rising Action:** 시간 입력 단계에서 **"모름"·"대략만 알아"** 선택. 동의 문구 후 기본값(예: 12:00) 또는 "시간 제외" 모드로 진행. 제한된 해석임을 안내.
- **Climax:** 시간이 없어도 **무료 4주제**는 최소한의 형태로 제공. "완전한 건 아니지만, 대충은 느껴진다" 수준.
- **Resolution:** 이탈을 줄이고, "시간 알게 되면 다시 입력해서 풀 버전 보기" 유도.

**요구사항:** "시간 모름" 옵션, 제한 모드/기본값, 관련 안내 문구, (선택) 나중에 시간 보완 입력 플로우.

## Innovation & Novel Patterns

### Detected Innovation Areas

- **결과 UX:** "한 장 = 심볼 1개 + 해당 주제 해석 + [ 다음 카드 ]". 기존 네이탈·운세의 "긴 글·스크롤"을 **한 화면에 한 섹션(주제)에 대한 내용**으로 전환.
- **가정 전환:** "결과는 길어야 한다" → 짧게, "생년월일·시간·장소 필수" → 최소·단계·"시간 모름" 허용.
- **조합:** **네이탈/출생차트 + LLM + 구조화 카드**. LLM 출력을 고정 포맷으로 변환해 "GPT보다 나를 잘 안다" **체감**을 노림.
- **참조 이식:** 토스·당근·듀오의 "한 화면 한 정보, 카드, 가벼운 성취"를 운세 도메인에 적용.

### Market Context & Competitive Landscape

- **기존:** 긴 해석, 스크롤, 어둡고 신비로운 톤, 입력 부담.
- **natalchart:** 한 화면에 한 섹션(주제) 분량의 카드, 밝은 톤, 스텝당 한 입력, "시간 모름" 대응. **기능보다 UI/UX·체감**으로 구분.

### Validation Approach

- **재방문률**, "나를 잘 안다" 체감(피드백/설문), **무료→유료 전환률**.
- 무료 4주제만으로 "다시 보고 싶다"가 나오는지, 유료 잠금 노출 후 전환이 일어나는지로 검증.

### Risk Mitigation

- **"LLM이 틀렸다" 인식** → 출생차트·주제 구조를 잘 넣고, **한 장 포맷·톤 가이드**로 일관성 유지.
- **체감이 안 맞으면 이탈** → 무료 4주제 품질·훅을 올리고, 유료 전 "이해·기대" 문구 설계.

## web_app Specific Requirements

### Project-Type Overview

**Next.js**, **SSR** 기반. 랜딩·입력·(선택) 카드 페이지는 서버 렌더, 카드 전환·LLM 로딩 등은 클라이언트. **모바일 우선** 반응형. 출생 정보 입력 → 출생차트 계산·LLM 호출·한 장 카드 노출을 **실시간에 가깝게** 처리. **SEO·OG·공유 URL**은 SSR로 대응. 접근성은 MVP에서 기본(포커스·대비·터치), WCAG AA는 Growth.

### Technical Architecture Considerations

- **Framework:** Next.js (App Router 또는 Pages Router).
- **Rendering:** SSR. 출생차트·LLM 호출은 서버에서 수행해 API 키·비즈니스 로직 보호. 카드 전환·`[ 다음 카드 ]`·로딩 UI는 클라이언트.
- **모바일 우선:** 카드·입력폼 반응형. iOS Safari, Android Chrome, 데스크톱(Chrome, Safari, Edge) 지원.

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

- **MVP Approach:** Experience MVP + Revenue MVP. 무료 체험으로 가치 체감, 유료로 전환·수익 검증.
- **"유용하다" 최소:** 무료 4주제 한 장 카드까지 노출, "맞는 말이다" 체감.
- **"잠재력 있다" 최소:** 재방문 가능성 + 유료 잠금 노출 후 결제 플로우 동작.
- **검증:** 재방문률, 무료→유료 전환률, (선택) "나를 잘 안다" 피드백.

### MVP Feature Set (Phase 1)

**지원 여정:** 첫 방문(무료 4주제), 유료 전환(잠금·결제·해금), "시간 모름" 예외.

**Must-Have:** 온보딩(생년월일·시간·장소, "시간 모름"), 출생차트 계산, LLM 연동·구조화·한 장 카드, 무료 4주제 카드·`[ 다음 카드 ]`, 유료 10주제 잠금·결제/구독·구매 이력·해금, 한 장 카드 UI(모바일 우선), Next.js SSR(랜딩·입력·OG), 계정·세션(결제·재방문 연계).

**Nice-to-Have (MVP):** "오늘의 한 장"·재노출, 카드/결과 URL 공유(OG).

### Post-MVP Features

- **Phase 2 (Growth):** 입력·"시간 모름" 고도화, 톤·문구 가이드, 이메일/푸시 재방문·유료 유도, WCAG AA, 성능·LCP 목표.
- **Phase 3 (Vision):** 1회 결제·구독·후원·UGC, 장기 데이터 개인화, 주제·과금 옵션 확장.

### Risk Mitigation Strategy

- **Technical:** LLM 품질·구조화 → 프롬프트·파싱·폴백. 출생차트 → 검증·테스트.
- **Market:** 재방문·전환 부족 → 무료 4주제·훅 강화, 유료 노출·가격 실험.
- **Resource:** 인력 부족 → 결제 외부 위임, Admin/Support 최소, MVP에서 해금 주제 일부만 단계 적용.

## Functional Requirements

### 1. 온보딩·입력 (Onboarding & Input)

- FR1: 사용자는 생년월일을 입력할 수 있다.
- FR2: 사용자는 출생 시간을 입력할 수 있다.
- FR3: 사용자는 출생 시간을 모른다는 것을 선택할 수 있다.
- FR4: 사용자는 출생 장소를 입력할 수 있다.
- FR5: 사용자는 입력 단계를 한 번에 하나씩 진행할 수 있다.
- FR6: 시스템은 입력된 출생 정보를 출생차트 계산 및 해석 생성에 사용할 수 있다.

### 2. 출생차트·해석 생성 (Chart & Interpretation)

- FR7: 시스템은 생년월일·시간·장소로 출생(네이탈) 차트를 계산할 수 있다.
- FR8: 시스템은 출생차트와 주제 컨텍스트를 LLM에 전달해 해석을 요청할 수 있다.
- FR9: 시스템은 LLM 응답을 구조화된 카드 형식(심볼, 1~2문장, CTA)으로 변환할 수 있다.
- FR10: 시스템은 14개 주제 각각에 대해 한 장 카드 해석을 생성할 수 있다.
- FR11: 사용자는 카드 생성·LLM 호출 대기 중 로딩·진행 상태를 인지할 수 있다.
- FR12: 시스템은 출생 시간을 모를 때 제한된 또는 근사 해석을 제공할 수 있다.

### 3. 카드 표시·네비게이션 (Card Display & Navigation)

- FR13: 사용자는 한 번에 한 장의 카드만 볼 수 있다.
- FR14: 사용자는 각 카드에서 심볼, 핵심 1~2문장, CTA를 볼 수 있다.
- FR15: 사용자는 다음 카드로 이동할 수 있다(예: [ 다음 카드 ]).
- FR16: 사용자는 무료 4주제 카드를 순서대로 볼 수 있다.
- FR17: 사용자는 해금된 유료 주제 카드를 볼 수 있다.

### 4. 무료/유료·접근 제어 (Access Control)

- FR18: 시스템은 4개 무료 주제를 결제 없이 노출할 수 있다.
- FR19: 시스템은 미구독·미구매 시 10개 유료 주제를 잠금(자물쇠·블러 등) 상태로 노출할 수 있다.
- FR20: 시스템은 구매·구독 여부에 따라 유료 주제 카드 접근을 제한할 수 있다.
- FR21: 사용자는 어떤 주제가 무료이고 어떤 주제가 유료(잠금)인지 구분할 수 있다.

### 5. 결제·구독 (Payment & Subscription)

- FR22: 사용자는 유료 주제 해금을 위해 결제 또는 구독을 시작할 수 있다.
- FR23: 사용자는 결제 또는 구독 플로우를 완료할 수 있다.
- FR24: 시스템은 구매·구독 이력을 기록하고 사용자와 연결할 수 있다.
- FR25: 시스템은 결제 완료 후 해당 유료 주제(들)를 해금할 수 있다.
- FR26: 사용자는 결제·구독으로 해금할 유료 주제 또는 패키지를 선택할 수 있다.

### 6. 계정·세션·재방문 (Account, Session & Revisit)

- FR27: 사용자는 방문 간에 식별 가능한 세션 또는 계정을 가질 수 있다.
- FR28: 시스템은 구매 이력을 사용자와 연결해 보관할 수 있다.
- FR29: 사용자는 재방문 시 이전에 해금한 유료 주제 카드를 다시 볼 수 있다.
- FR30: 사용자는 결제 또는 접근 유지가 필요할 때 로그인하거나 계정을 만들 수 있다.

### 7. 랜딩·SEO (Landing & SEO)

- FR31: 사용자는 서비스 진입을 위한 랜딩(메인/온보딩) 페이지에 도달할 수 있다.
- FR32: 시스템은 랜딩 및 핵심 페이지에 메타·OG 태그를 제공해 SEO 및 SNS 공유 미리보기를 지원할 수 있다.

## Non-Functional Requirements

### Performance

- **NFR-P1:** 랜딩·입력 첫 화면은 LCP 2.5초 이내(또는 목표치)에 사용 가능해야 한다.
- **NFR-P2:** 카드 생성 요청(출생차트 계산·LLM 호출·구조화) 후, 사용자가 카드를 볼 수 있을 때까지 목표치 이내(예: 15초 이내)여야 하며, 그 전까지 로딩·진행 상태를 보여줘야 한다.
- **NFR-P3:** 카드 간 전환([ 다음 카드 ])은 1초 이내에 반영되어야 한다.

### Security

- **NFR-S1:** 출생 정보(생년월일·시간·장소)는 전송 시(TLS 등) 암호화되어야 한다.
- **NFR-S2:** 출생 정보·구매 이력은 인증된 사용자 본인이 조회·확인할 수 있어야 한다. 저장·전송 시 권한 없는 제3자 접근을 막기 위한 보안 조치(접근 제어, 전송 암호화 등)를 적용해야 한다.
- **NFR-S3:** LLM API 키·결제 관련 비밀·서버 전용 로직은 클라이언트에 노출되지 않아야 한다.
- **NFR-S4:** 인증된 사용자만 해당 사용자의 구매 이력·해금 주제에 접근할 수 있어야 한다.

### Accessibility

- **NFR-A1:** 입력 필드·카드·CTA는 키보드 포커스로 조작 가능해야 한다.
- **NFR-A2:** 터치 타겟(버튼·[ 다음 카드 ] 등)은 44×44px 이상이어야 한다.
- **NFR-A3:** (Growth) WCAG 2.1 AA 준수를 목표로 한다. MVP에서는 포커스·대비·터치 영역만 필수.

### Integration

- **NFR-I1:** LLM API 장애·타임아웃 시 사용자에게 에러 안내 또는 재시도 옵션을 제공해야 한다.
- **NFR-I2:** 결제 연동(PG) 장애 시 결제 플로우에서 명확한 안내와, 가능하면 재시도 경로를 제공해야 한다.

### Scalability (MVP 경량)

- **NFR-SC1:** (MVP) 동시 접속·일 사용자 수가 초기 목표 규모(예: 1,000 DAU)일 때, NFR-P1·P2·P3가 유지되어야 한다. 구체 용량·한도는 아키텍처 단계에서 정의.
